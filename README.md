# 工事進捗ダッシュボード

`index.html` をダブルクリックするとブラウザで開きます。インターネット接続やインストールは不要です。

初版の集計基準日は2026年7月10日、履行期限は2026年9月30日です。施工計画書がローカル利用可能になった後、計画工程を更新してください。

## 使用材料資料

同じ工事フォルダの `5.【材料関係】` にある製品カタログをもとに、材料ごとの「使い道」「現場での使い方」「使用前の確認」と、公開確認済みのカタログを `materials.html` に表示します。

2026年7月14日、材料原本44件を再分類し、全件がローカル利用可能であることを確認しました。材料承認資料は製品選定の内部根拠にのみ使い、一般カタログの全ページを確認して、個人情報・価格・取引情報を含まない8製品資料を公開用に複製しました。公開用PDFは文書メタデータも公開用へ置き換えています。

2026年7月15日、ブラウザ内蔵PDF表示に依存せず、ローカルのダブルクリック表示、GitHub Pages、iPhoneのいずれでも読めるよう、8製品・全21ページを画像化した専用ビューア `catalog.html` を追加しました。材料一覧の「カタログを見る」はPDFを直接開かず、該当製品の全ページを縦に並べたビューアへ移動します。各ページをタップすると画像単体で拡大できます。表示画像は確認済みの公開用PDFから生成し、元の材料原本と公開用PDFは変更していません。

同日、利用者による材料一覧・カタログ表示の確認と公開承認を受けました。材料機能は7月15日までの進捗実績とともにGitHub Pagesへ公開し、以後は週報更新時に公開用ファイルとリンクを再検査します。

掲載対象は、エポティ、Vフロン#100Hスマイル上塗 Re、ケミカルアンカー R-16N、RIS211E、プレタスコン TYPE-LS500、NSポリマーミックス#10、NSハイフレックス HF-1000、共英製鋼の鉄筋です。請求書、納品書、見積書、出荷案内書など23件の取引書類、承認資料3件、価格を含む製品ページ、承認資料で対象を特定できない資料は掲載していません。コーキング剤と六角ボルトは、価格を含まない対象製品の資料を確認できるまで保留します。NSゼロヨンとそのSDSは資料を確認済みですが、今回の材料承認資料に対象製品として記載がないため掲載していません。

AI要約だけで施工判断を行わせず、原本カタログ、製品表示、SDS、承認条件を優先します。公開前の確認手順と判断基準は [MATERIALS_IMPLEMENTATION_HANDOFF.md](MATERIALS_IMPLEMENTATION_HANDOFF.md) に記録しています。

材料一覧は、開いたカードと同じ行の閉じたカードが空白のまま引き伸ばされないよう、PC・スマートフォンとも1列のアコーディオンとしています。展開した説明はPCで3列、スマートフォンで1列に切り替わります。

カタログを差し替えるときは、公開可否と全ページを再確認したうえで `materials/catalogs/` の公開用PDFを更新し、同じページ数の画像を `materials/pages/<材料ID>/page-1.jpg` から連番で生成します。`data.js` の `catalogPages` を実ファイルと一致させ、PDFのページ数、画像数、全画像の描画内容を確認してください。

## 更新手順

1. このフォルダと同じ `4.【作業報告書】` 配下にある月別週報を確認する。
2. 週報の「実施」欄に記載された日だけを確定実績として `data.js` の `recent`、`phases`、`calendarRanges` へ反映する。
3. 基準日より後の「予定」欄は `upcoming` と、`status: "planned"` の工程・カレンダーへ反映する。予定は必ず画面上でも「予定」と表示し、実績と混同させない。
4. 予定日が過ぎても、自動的に実績へ変更しない。更新された週報の「実施」欄を確認してから、該当予定を削除または修正し、実績表記へ置き換える。未確認の場合は予定のまま残すか、掲載から外す。
5. `project.asOf` は実施を確認できた最新日、`lastActivity` は最新の実施日、`updatedAt` はサイトを更新した日本時間、`sourceFiles` は実績・予定の確認に使った週報数とする。
6. 個人名、会社名、連絡先、週報原文、詳細な内部打合せ、数量、型番は公開しない。
7. ギャラリーへ追加する写真は、顔・車両番号・連絡先などが写っていない代表カットだけを選ぶ。元写真は変更せず、長辺1600px程度に縮小した公開用JPEGを `gallery` へ保存し、`data.js` の `gallery` に日付・工程・説明・代替テキストを追記する。
   - 日々の細かな進捗は追加せず、新しい工程の開始、主要工程の完了、施工状態や外観の明確な変化、工事再開など、閲覧者が工程変化を理解できる節目に限定する。
   - 同じ工程・同じ状態の類似写真は増やさず、既存写真より変化が分かりやすい場合は追加ではなく差し替えも検討する。
   - 写真追加後もスケジュールが主役であることを守り、ギャラリーはページ最下部の切替表示を維持する。
8. `recent` は最新日を先頭にした順序を維持して追記し、過去の記録を表示上の都合で削除しない。「直近の作業」はカード内スクロールでおおむね6日分を表示し、PCでは左隣のカレンダーカードと同じ高さに自動調整する。スマートフォンでは固定同期せず、読みやすい上限高の中でスクロール表示する。
9. `node --check data.js`、`node --check app.js`、`node --check materials.js`、`node --check catalog.js`、ブラウザ表示、Git差分を確認する。利用者から承認がある場合に限り、日本語のコミットメッセージで `origin` の作業ブランチへプッシュする。

GitHub Pagesの公開対象は `.github/workflows/deploy-pages.yml` の「Prepare public files」で指定する。画像などの公開用フォルダを追加した場合は、同じ工程で `_site` へコピーする設定も必ず追加し、公開URLでファイルがHTTP 200になることを確認する。

HTML・CSS・JavaScript・画像の構成を変えたときは、`index.html` のCSSとJavaScriptのURLへ同じ更新番号（例：`?v=20260712-1126`）を付け替える。これにより、公開直後に古いJavaScriptと新しいHTMLが混在して画面生成が止まることを防ぐ。JavaScript側も、削除済みまたは未追加の任意要素を参照して全体停止しないよう存在確認を行う。

## 2026年7月12日 表示停止事故の記録

### 症状

- 工程表は月目盛りだけになり、工程行が表示されなかった。
- カレンダー、直近の作業、今後の予定、写真ギャラリーが空になった。
- 最終更新日時も空欄になった。

### 原因

「確認待ち事項」カードをHTMLから削除した直後、ブラウザには削除前の `app.js` がキャッシュされたまま残る場合があった。古いJavaScriptが削除済みの `data-note` 要素へ書き込もうとして例外となり、それ以降の画面生成処理がすべて停止した。新しいHTMLと古いJavaScriptが混在したことが直接原因である。

### 実施した復旧

1. CSS、`data.js`、`app.js` のURLへ共通の更新番号を付け、古いキャッシュと区別した。
2. 要素への書き込みを `setText` にまとめ、対象要素が存在しない場合は処理を継続するよう変更した。
3. `gallery` と `upcoming` がない古いデータを読み込んでも、空配列として扱い全体停止しないよう変更した。
4. GitHub Pagesの公開完了後、公開サイトで工程8行、カレンダー19件、直近作業9件、今後予定8件、写真7件が生成されることを確認した。

### 今後の必須確認

- HTMLの要素を追加・削除した場合は、JavaScript側の参照も同時に確認する。
- 更新ごとにCSS・JavaScriptの更新番号を揃えて変更する。
- ローカル表示だけで完了とせず、GitHub Pages公開後のURLで各一覧の件数とブラウザエラーを確認する。
- 一部分の不具合で画面全体が空にならないよう、任意データと任意要素には存在確認を入れる。
